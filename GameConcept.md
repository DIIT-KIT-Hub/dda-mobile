## Концепт гри Bug Detective

**Bug Detective** — це 2D-гра-головоломка для Android у нуарному стилі (чорно-біле місто з кольоровими акцентами), у якій гравець виступає у ролі **детектива з відділу програмних розслідувань** і розкриває злочини, спричинені багами у програмному коді.

Гра одночасно:
- **Розважає** — нуарний наратив, NPC, прогрес кейсів, ачівменти.
- **Навчає** — кожен кейс відпрацьовує конкретну техніку налагодження (debugging skill): локалізація багу, класифікація, формулювання гіпотези, виправлення.
- **Адаптується** — модель машинного навчання (DDA) на пристрої аналізує поведінку гравця і автоматично підлаштовує складність наступного кейса.

### Сетинг і атмосфера

- **Місто Bytetown** — нуарний мегаполіс, де компанії, лікарні, банки і транспорт залежать від програмного коду.
- **Гравець** — детектив-початківець у відділі CIB (Code Investigation Bureau).
- **Сюжет** — серія розслідувань: десь зламалася система → постраждали люди → детектив прилітає на «місце злочину» (фрагмент коду + опис інциденту) і має знайти баг, що став причиною.
- **Стилістика візуалу** — пласкі векторні ілюстрації, темна палітра з акцентами червоного/жовтого (виділення коду, підсвітка багу). Шрифти моноспейсні для коду, рукописні для нотаток детектива.

### Ігровий цикл

Один кейс = один «раунд» гри. Структура кейсу:

1. **Брифінг (10-30 с)** — NPC коротко розповідає, що сталося. Наприклад: *«О третій ранку зник переказ заробітної плати усім співробітникам із табельним номером, який ділиться на 256. Програміст клянеться, що нічого не міняв».*
2. **Огляд місця злочину (5-15 с)** — на екрані з'являється фрагмент коду (10-40 рядків Kotlin або псевдокоду) і скрін зі стек-трейсом / логами / результатом, який отримав постраждалий.
3. **Розслідування (60-300 с)** — гравець виконує три послідовні дії:
   - **Локалізація**: тапає на рядок, де, на його думку, баг.
   - **Класифікація**: обирає категорію багу зі списку (off-by-one, null dereference, race condition, integer overflow, тощо).
   - **Виправлення**: обирає правильне виправлення з 3-4 варіантів (multiple choice) АБО (на просунутих рівнях) вписує патч у текстове поле.
4. **Розв'язка (15-30 с)** — анімація: код «оживає», NPC дякує детективу, гравець отримує бали досвіду (XP), бейдж за категорію бага, можливо — підказку до наступного кейса.
5. **DDA-такт (фоном)** — після кожного кейса плагін збирає телеметрію (час, точність на кожному кроці, кількість підказок, класифікаційна правильність) → запускає TFLite-модель → отримує `delta` ∈ [-1, +1] → застосовує до параметрів наступного кейса (рівень складності бага, час на розв'язання, кількість показуваних підказок).

Один сеанс гри = 5-10 кейсів × 3-5 хвилин = 15-50 хв.

### Категорії багів

Кожен кейс належить до однієї з категорій. Всі категорії впорядковані за рівнем складності (від 1 до 5) — це і є основний параметр, який крутить DDA.

| Рівень | Категорія | Приклад |
|--------|-----------|---------|
| 1 | Синтаксичні помилки | Пропущена дужка, друкарська помилка у назві змінної |
| 1 | Друкарські помилки в логіці | `=` замість `==`, `||` замість `&&` |
| 2 | Off-by-one | Цикл `< n` замість `<= n` або навпаки |
| 2 | Null / Optional dereference | `obj!!.field` коли obj дорівнює null |
| 2 | Невірна умова виходу | Цикл, який ніколи не зупиняється |
| 3 | Side effects | Функція мутує переданий список, хоч не повинна |
| 3 | Scope / shadowing | Внутрішня змінна з тим самим іменем приховує зовнішню |
| 3 | Type coercion | Цілочисельний поділ замість дробового |
| 4 | Race conditions | Дві корутини читають/пишуть спільну змінну без синхронізації |
| 4 | Deadlock | Два потоки чекають один одного на блокуваннях |
| 4 | Integer overflow | Лічильник int переповнюється на великих значеннях |
| 5 | Memory leaks | Listener не зніметься з активності → витік |
| 5 | Performance | N+1 запит до бази даних, нелогічна квадратична складність |
| 5 | Security | SQL-ін'єкція, hardcoded credentials, broken authentication |

### Прогрес і наратив

- **Розділи** — 4-5 сюжетних арок, кожна об'єднує 8-12 кейсів спільним сетингом (наприклад: «Справа банку Bytetown», «Зловмисний планувальник», «Витік пам'яті в лікарні»).
- **NPC-наставник** — старший детектив Інспектор Логос — пояснює нові категорії багів і дає підказки.
- **NPC-злочинці** — карикатурні «архетипи» (Лінивий програміст, Менеджер-який-вимагає-фічі-вчора, Привид-старий-код), за якими стоять реальні причини багів.
- **Бейджі і колекція** — за кожну категорію окрема медаль.

### Що адаптує DDA

DDA-модель повертає скалярний `delta` ∈ [-1, +1]. Контролер DDA транслює це у конкретні параметри наступного кейса:

- **Рівень складності бага** (1-5) — головний параметр.
- **Кількість рядків коду** в фрагменті (10/20/30/40).
- **Час на розв'язання** (60-300 с).
- **Кількість безкоштовних підказок** (0-3).
- **Балансування категорій** — модель відстежує, у яких категоріях гравець стабільно слабкий, і частіше дає кейси саме звідти.

DDA НЕ адаптує наратив, графіку, складність *тексту* діалогів — тільки навчальний контент.
